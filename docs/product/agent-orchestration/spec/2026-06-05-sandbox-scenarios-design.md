# 沙箱场景设计：工作流与聊天

> 文档定位：在《[沙箱生命周期设计](2026-06-05-sandbox-lifecycle-design.md)》（沙箱通用机制——常驻、空闲回收、skill/MCP 装配）之上，分别阐明**两种触发场景**下沙箱如何使用：① 工作流场景（模板编排的 workflow run）；② 聊天场景（即席对话直接触发任务，不经工作流）。第 3 节给出场景对照，第 4 节汇总对 **Agent Core 的能力要求**，区分「沙箱内执行」与「非沙箱（纯 LLM）执行」。
>
> 设计目标（按优先级）：① 大规模用户并发无压力；② 资源利用合理；③ 用户体验流畅。

## 0. 概览

| 维度 | 工作流场景（§1） | 聊天场景（§2） |
|---|---|---|
| 触发 | Agent-Management 组装 AgentFlow → `POST /runs` | 用户在聊天页发消息，不经工作流 |
| 任务结构 | 预定义 DAG，多 step | 无 DAG，连续对话 |
| 沙箱寻址键 | `runId`（run 级） | `conversationId`（会话级） |
| 沙箱触发 | run 启动即创建 | 惰性——仅当某轮需要执行时 |
| 生命周期 | 随 run 常驻，终态销毁 | 四态分层 NONE→WARM→FROZEN→RECLAIMED |
| 复杂任务承载 | 本身就是 | 派生工作流 run 去跑 |
| 是否经编排层 | 是（全程） | 快路径不经，慢路径派生 run 才经 |

一句话：**工作流是"先有结构、再执行"；聊天是"先对话、按需升级出结构"。** 底层共用同一套沙箱机制，差别在寻址键、创建时机、回收策略。

---

## 1. 工作流场景：run 级常驻沙箱

> 完整机制见《沙箱生命周期设计》，此处仅作要点回顾与对照基线。

- **沙箱 = run 级常驻**：首个 step 启动时按 `workspaceRef` find-or-create，贯穿整个 run，终态时销毁、释放 lease。
- **step = 沙箱内 session**：文件全 run 共享，上游改动下游直接可见，无需跨沙箱传递。
- **进程可死、对话可恢复**：进程退出后沙箱与文件、对话上下文都在，续聊在同沙箱内 `--resume`。
- **skill/MCP 按 step 装配**：每个 step 的 session 启动时按本 step 引用装配，相邻同引用可复用 MCP server。
- **挂起成本**：`requiresConfirmation` 挂起期间沙箱保活（可休眠）。

特征：生命周期清晰、绑定明确（绑 run、绑 workspace），结构预定义，资源开销可预测，无需聊天那套惰性/分层。

---

## 2. 聊天场景：会话级惰性沙箱 + 快慢双路

聊天任务有三个特征，决定它**不能套用 workflow run 模型**：无预定义 DAG（一句话是一个意图）、连续对话（会追问改需求）、简繁混杂（「看下这函数」和「重构整个模块」走同一输入框）。硬把每句话包成单 step 的 AgentFlow 很别扭：每句话一个 run/沙箱、连续性全靠 resume 拼、run 概念对用户是纯负担。所以聊天需要一条**独立的轻量执行路径**。

### 2.1 沙箱按会话寻址，惰性创建、四态分层

两条主线：

**(A) 沙箱按会话（conversation）寻址，不按"任务"。** 一个聊天窗口的多轮天然共享上下文和文件，所以一个会话对应一个沙箱、一个持续 session。复用 run 级常驻思想，寻址键从 `runId` 换成 `conversationId`。

**(B) 沙箱惰性创建 + 四态分层。** 大规模下绝大多数轮次（纯问答）不碰文件、不调工具，**不该占用任何沙箱**。沙箱只在真正需要执行时才 provision，空闲后逐级回收。这是资源能否撑住规模的关键。

把"会话"与"沙箱"解耦成两个生命周期——会话是廉价持久对象（只存消息），沙箱是按需挂载的昂贵资源：

```text
会话（durable，廉价，只存消息历史 + sandbox 指针）
  └─ 沙箱挂载状态（昂贵，按需）：
       NONE ──首次需执行──► WARM ──空闲 idle-soft──► FROZEN ──空闲 idle-hard──► RECLAIMED
        ▲                     │                                                  │
        │                     └─────────── 用户新消息需执行 ◄────────────────────┘
        └─ 纯问答轮次：全程停在 NONE，不占沙箱
```

| 态 | 含义 | 占用 | 进入条件 | 服务目标 |
|---|---|---|---|---|
| **NONE** | 无沙箱，纯 LLM 对话 | 仅会话记录（KB 级） | 会话初始 / 当前轮无需执行 | ①②|
| **WARM** | 沙箱存活，可立即执行 | 完整沙箱（容器+卷） | 某轮需碰文件/工具 | ③（秒级响应）|
| **FROZEN** | 沙箱冻结，文件保留、不占算力 | 仅存储（卷快照） | WARM 后空闲超 idle-soft（建议 10min）| ② |
| **RECLAIMED** | 沙箱销毁，对话与文件外置持久 | 仅外置存储 | FROZEN 后空闲超 idle-hard（建议 1h），或会话关闭 | ①② |

- **NONE→WARM**：某轮被判定需要执行（见 §2.3）时才惰性创建沙箱。
- **WARM→FROZEN**：冻结进程/容器但保留 workspace 卷快照，再来消息时解冻，比冷启快。
- **FROZEN→RECLAIMED**：彻底销毁，对话上下文与文件改动外置持久化（同生命周期文档的 sessionRef 落盘）；再来消息按冷启 + `--resume` 恢复。
- 用户感知的"聊天窗口"始终在；底下沙箱在各态流转对用户透明，只体现为响应延迟差异。

> 这套分层让"一万个开着的聊天窗口"的真实成本逼近"当前正在执行的少数会话"，而非一万个沙箱。**这是支撑大规模的核心。**

### 2.2 快慢双路

同一输入框，按任务量级分流：

**快路径（简单任务，默认，绝大多数）**：在会话沙箱里直接跑（NONE 则惰性升 WARM），就是多一轮对话。不开 run、不走编排、不进状态机。问答、读代码、改几行、跑个命令——都在会话沙箱里连续做，文件多轮共享。纯问答连沙箱都不要，停在 NONE。

**慢路径（复杂任务，少数，派生正式 run）**：当一句话其实是需长时间、多步骤、要审查的大活，**不在聊天沙箱里硬跑**，而是从聊天**派生一个工作流 run**——run 在自己独立的 run 级沙箱里跑重活（即 §1 那套），不阻塞聊天；聊天会话变成 run 的**遥控器 + 观察窗**（看进度、做确认、给反馈，事件流回聊天展示）。复杂任务由此自动获得 run 的全套能力：状态机、重试、审查 gate、可恢复、可观测。

### 2.3 怎么判断简单 vs 复杂：三层递进，不靠一次性预判

**核心原则：不要在用户发消息那一刻用一个分类器一次性判定"简单还是复杂"。** 同一句"修下登录 bug"，可能改一行就好，也可能牵出整个鉴权模块——**复杂度在执行前无法准确预知**。预判分类器要么过度保守（动不动建任务、体验差），要么漏判（大活塞进聊天沙箱跑到一半卡住）。

所以用三层递进，从"廉价预判"到"执行中暴露"，而不是赌一次猜对：

**第一层 · 预判分流（快，只做粗筛）。** 发消息瞬间做一次轻量意图判断，目的不是精确分类，而是**只筛两个极端**：
- 明显纯问答（"这段代码啥意思"）→ 停在 NONE，纯 LLM 回答，不要沙箱；
- 明显大工程（"把整个项目从 JS 迁到 TS"）→ 直接提议建 run；
- **中间地带（绝大多数）→ 一律先走快路径**，别纠结。

中间地带默认乐观走快路径，因为会话沙箱本就常驻，先试代价很低，把判断推迟到执行中（那时信息多得多）。这层不需重模型，几个信号就够：动词（解释/问 vs 重构/迁移/批量）、是否提"整个/所有/全部"、是否涉多文件、是否要跑测试/构建/部署。规则 + 小模型兜底即可。

**第二层 · 执行中的复杂度护栏（真正可靠的判据）。** 核心——**让任务自己暴露复杂度，而非预测它**。快路径执行时设几条护栏，**任意一条触发就当场提议升级为 run**：

| 护栏信号 | 阈值（示例） | 为何是复杂度硬证据 |
|---|---|---|
| 工具调用/轮次数 | 本轮自主调用工具超 N 次（如 15）| 简单任务几步就完，停不下来说明是大活 |
| 单轮耗时 | 超 T 秒（如 60s）| 超出用户在聊天里能忍的等待 |
| 文件改动范围 | 改动文件超 M（如 10）或触及核心目录 | 大范围改动需审查，不该在聊天里悄悄发生 |
| 触发审查规则 | 命中危险操作/高风险路径（改鉴权、删数据、动 CI）| 必须进 run 的审查 gate |
| token/成本 | 单轮消耗超预算 | 成本失控信号 |

这些都是执行中才拿得到的真实信号，比预判准得多。触发时不硬切，而是给用户选择："这个改动比较大（已动 12 个文件），转成后台任务跑、跑完给你审查，可以吗？" 护栏的另一重作用是**兜底**：就算预判漏了大活，护栏也会在它跑失控前拦住，不拖垮会话沙箱——这是规模与体验的安全网。

**第三层 · 用户显式意图（最高优先级）。** 用户说了算，覆盖前两层：直接说"建个任务跑"→ 直接 run；界面给轻量开关主动标"大任务"；对升级提议说"不用就在这改"→ 尊重并留在快路径（但**审查规则那条护栏不可被覆盖**，安全底线）。

> 一句话：**预判只筛极端、中间地带乐观快跑、用执行中真实护栏定夺升级、用户意图永远优先。** 复杂度不是预测出来的，是任务跑起来后自己暴露的。

### 2.4 沙箱是否绑定 workspace/repo（惰性绑定）

- 会话可关联一个项目上下文（用户在哪个项目里发起聊天）。绑定项目 = 可访问该项目 repo 与连接器，能改代码；
- **repo 惰性 clone**：NONE→WARM 首次创建沙箱时才按需 clone，纯问答阶段不 clone；
- 无项目上下文的聊天 = 空沙箱，只支持问答与轻量文件操作，不碰 repo；
- "简单任务能干多重"取决于是否绑项目：绑了能在快路径直接改代码、跑测试；没绑就是纯助手。

### 2.5 派生 run 时的上下文交接

慢路径派生 run 时把聊天上下文带过去，否则 run 从零开始、用户得重述：

- **对话摘要注入**：把会话至今的关键意图、约束、已达成结论摘要后，作为派生 run 首个 step 的 prompt 上下文（或 `AgentFlow.variables`）；
- **文件状态交接**：若聊天沙箱已改 repo，run 需基于这些改动继续——会话沙箱先 commit、run 沙箱按该 commit checkout（**跨沙箱传递，用 git commit 作硬证据**，因聊天沙箱与 run 沙箱是两个沙箱、不共享卷）；
- **会话↔run 双向链接**：会话记录 `runId`，run 事件回流到会话展示；会话里的确认/反馈转成对 run 的 confirm/continue 调用。

### 2.6 Skill 与 MCP 加载

与 run 沙箱一致的按需装配，触发点不同：

- 会话关联一组允许的 agent 能力（项目/用户级 allowlist）；
- NONE→WARM 创建沙箱时按会话关联的 `skillSnapshotRefs / mcpSnapshotRefs` 装配；
- 会话存活期内能力集相对稳定（不像 run 每 step 切配置），MCP server 可长期复用，FROZEN 时随沙箱冻结、解冻时恢复；
- 凭证按最小权限注入短期令牌，RECLAIMED 时回收。

## 3. 两场景对照与三要求落点

| 要求 | 工作流场景 | 聊天场景 |
|---|---|---|
| **大规模** | run 数可控、生命周期明确 | 沙箱惰性创建（纯问答停 NONE）；四态分层让成本≈活跃会话数；复杂任务派生独立 run 不占聊天沙箱 |
| **资源合理** | 终态即销毁、释放 lease | WARM→FROZEN→RECLAIMED 逐级释放；冻结留卷快照避免重复 clone；MCP 随沙箱冻结/复用 |
| **体验好** | 状态机/重试/审查保障可靠交付 | 快路径秒级、对话连续、文件共享；沙箱态流转透明；复杂任务一键升级后台跑、带上下文不重述 |

## 4. 对 Agent Core 的能力要求总结

两场景对 Agent Core 提出的能力，按「**沙箱内执行**」与「**非沙箱执行**」两类汇总。这是本文给 Agent Core 的接口/实现需求清单。

### 4.1 非沙箱能力（纯 LLM，不分配沙箱）

仅聊天场景需要，是大规模省资源的前提：

1. **无沙箱对话执行**：对纯问答轮次，只做 LLM 推理（可叠加只读上下文如已索引的代码片段），**不创建任何沙箱**，直接流式返回；
2. **轮次级意图预判**：提供轻量判断能力，把轮次粗分为「纯问答 / 需执行 / 明显大活」三类（规则 + 小模型即可），用于决定是否升 WARM 或提议派生 run；
3. **无状态低延迟**：非沙箱轮次应可水平扩展、无沙箱绑定，支撑高并发问答。

### 4.2 沙箱内执行能力（两场景共用）

1. **沙箱 find-or-create（按键寻址）**：支持按 `runId`（工作流）或 `conversationId`（聊天）find-or-create 沙箱；命中复用、未命中创建；
2. **沙箱常驻与多 session**：单沙箱承载多个 session（工作流的多 step、聊天的多轮），session 间文件共享、对话上下文隔离；
3. **进程可死 + 对话恢复**：对话上下文落盘（键 = sessionRef），进程退出不丢；支持 `--resume <sessionRef>` 重载续聊；
4. **按需装配 skill/MCP/知识库**：按传入引用拉取装配，支持相邻同引用复用 MCP server；凭证按最小权限注入短期令牌、用后回收；
5. **workspace 挂载与 repo 惰性 clone**：挂载持久卷；首次需要时才 clone/checkout repo；
6. **事件回报**：生命周期事件 + Agent 执行过程事件（thinking/message/tool_*/std*），统一协议上报。

### 4.3 仅聊天场景需要的沙箱能力

1. **沙箱冻结/解冻（FROZEN）**：暂停进程/容器、保留卷快照、停止计费算力；解冻快于冷启。实现形态（容器暂停 / 卷快照 / CRIU checkpoint）由 Agent Core 定（见 §5#3）；
2. **空闲驱动的状态流转**：依据 idle-soft / idle-hard 触发 WARM→FROZEN→RECLAIMED；保活期间继续 heartbeat；
3. **执行中复杂度信号上报**：把工具调用数、耗时、文件改动范围、命中审查规则等信号实时暴露，供上层护栏判定是否提议升级（见 §2.3 第二层）；
4. **跨沙箱状态交接支持**：支持聊天沙箱 commit、run 沙箱按 commit checkout，承接慢路径派生（见 §2.5）。

### 4.4 编排层（Agent-Orchestration）边界

- **工作流场景**：全程经编排层，沿用现有 run/step/attempt 状态机与 StartAttempt 契约（`workspaceRef + resumeFromSessionRef`，不加 sandboxRef）；
- **聊天快路径**：建议**直连 Agent Core 会话执行接口，不经编排层**，避免轻量交互被编排开销拖累；
- **聊天慢路径**：派生 run 后回到编排层标准流程。

## 5. 待确认 / 后续

| # | 议题 | 说明 |
|---|---|---|
| 1 | idle-soft / idle-hard 阈值 | WARM→FROZEN（建议 10min）、FROZEN→RECLAIMED（建议 1h）实测调优；可按负载动态调整 |
| 2 | 预判与护栏阈值 | 第一层意图判断的信号集；第二层护栏各阈值（N/T/M）的实测标定与误判回退 |
| 3 | FROZEN 实现形态 | 容器暂停 vs 卷快照 vs CRIU checkpoint，取决于执行后端能力 |
| 4 | 会话沙箱并发 | 单会话是否允许同时多执行轮（建议串行，与 run 串行单飞一致）；多设备/多标签页打开同一会话的处理 |
| 5 | 聊天快路径 API 边界 | 直连 Agent Core 的会话执行接口定义；与编排层职责切分 |
| 6 | 配额与成本归属 | 聊天沙箱资源配额、按用户/项目并发上限，防单用户开大量会话耗尽资源 |

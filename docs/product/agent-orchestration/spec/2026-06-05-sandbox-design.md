# 沙箱设计：生命周期、工作流与聊天场景

> 文档定位：在 Agent-Orchestration / Agent Core 架构下，统一阐明**沙箱（Sandbox）的设计**——核心模型、生命周期，以及两种触发场景（工作流 / 聊天）下的具体用法，最后汇总对 Agent Core 的能力要求。本文是对《orchestrator-agentflow-workflow-template-design》§8（Agent Core 接口）与 §8.8（对话上下文持久化）的细化与收口，不改变既有的 run/step/attempt 状态机与 StartAttempt 契约。
>
> 设计目标：① **规模化与资源效率**——大规模并发下资源用得合理、可弹性伸缩；② **用户体验流畅**——响应快、对话连续、过程透明；③ **安全与隔离**——沙箱作为租户/任务的隔离与安全边界。下文关键决策标注其服务的目标（效率 / 体验 / 安全）。
>
> 适用范围：Agent Core 的运行时实现约定。Agent-Orchestration 不感知沙箱实现细节，标注「编排层视角」的部分才是契约面，其余为 Agent Core 内部实现指引。

## 0. 总览

两种场景触发沙箱，共用同一套底层模型（§1），区别在沙箱归属、创建时机、回收策略：

| 维度 | 工作流场景（§2） | 聊天场景（§3） |
|---|---|---|
| 触发 | Agent-Management 组装 AgentFlow → `POST /runs` | 用户在聊天页发消息，不经工作流 |
| 任务结构 | 预定义 DAG，多 step | 无 DAG，连续对话 |
| 沙箱归属（Orchestration 持有映射） | run ↔ sandboxId | conversation ↔ sandboxId |
| 沙箱触发 | run 启动即创建 | 惰性——仅当某轮需要执行时 |
| 生命周期 | 随 run 常驻，终态销毁 | 四态分层 NONE→WARM→FROZEN→RECLAIMED |
| 空闲冻结（两场景共用同一套机制） | run 未终止但无活跃执行即冻结（挂起等确认、重试退避、step 间空档），需要时解冻 | 会话空闲即冻结（FROZEN），新消息需执行时解冻 |
| 复杂任务承载 | 本身就是 | 派生工作流 run 去跑 |
| 是否经编排层 | 是（全程） | 快路径不经，慢路径派生 run 才经 |

一句话：**工作流是"先有结构、再执行"；聊天是"先对话、按需升级出结构"。**

## 1. 沙箱核心模型（两场景共用）

### 1.1 三层解耦：Sandbox / Session / Attempt

沙箱设计的根基是把三件事拆开，各管一件、生命周期各不相同：

| 层 | 是什么 | 生命周期 | 共享范围 |
|---|---|---|---|
| **Sandbox（沙箱）** | 承载执行的隔离环境（容器/进程组 + 挂载的 workspace 卷） | 按场景：工作流=run 级，聊天=会话级 | 同一沙箱内所有 session 共享文件 |
| **Session（对话）** | 一次 Agent 对话上下文，键 = `sessionRef`，落在沙箱内 workspace 路径 | 工作流=step 级，聊天=一轮任务 | 同 session 的多个 attempt 共享 |
| **Attempt（执行轮次）** | 一轮执行 = 一次 `StartAttempt`；session 里的一个回合 | attempt 级：进程可死可重启 | 不共享，每轮独立 |

**关键性质：同一沙箱内多个 session 文件共享、对话上下文隔离。** 文件天然共享避免了"名义共享卷、实则各写各的临时盘"导致的改动丢失；对话隔离保证不同任务/step 的上下文不串味。

### 1.2 进程可死 + 对话恢复

沙箱与"进程可死"是不同维度：**沙箱管文件共享，进程/对话管对话续接。** 进程死的是沙箱里的执行进程，不是沙箱本身。

- 对话上下文是独立于执行进程的持久资源，落在沙箱内 workspace 约定路径（键 = `sessionRef`），进程退出不丢；
- 进程默认可退出；恢复靠对话上下文持久化 + `--resume <sessionRef>` 重载；
- 编排层对进程冷热**无感知**，续聊一律按"创建新 attempt"处理；冷热只影响延迟，不影响语义。

### 1.3 Skill / MCP / 知识库的按需装配

`StartAttempt` 携带本次执行所需的能力引用，Agent Core 在 session 启动时按引用拉取装配：

| 能力 | 引用字段 | 装配方式 |
|---|---|---|
| Agent 配置 | `agentSnapshotRef` | 按引用拉取快照，作为本 session 的 agent 配置 |
| Skill | `skillSnapshotRefs` | 按引用拉取 skill 内容，装载进本 session 可用技能集 |
| MCP Server | `mcpSnapshotRefs` | 拉取 MCP 配置，在沙箱内启动/注册 MCP server，注入 `mcpCredentialRefs` 短期凭证 |
| 知识库（文件形态） | `knowledgeBaseRefs` | 只读挂载到 workspace 内约定路径 |
| 知识库（检索服务形态） | `knowledgeBaseRefs` | 以检索服务（MCP 风格）注册，Agent 通过工具调用查询 |

装配通用约定：

- **能力集可动态切换**：同一沙箱内不同 session 的 skill/MCP 可能不同，不能假设一个沙箱终生只伺候一套配置；
- **MCP server 复用**：相邻 session 引用相同 MCP 快照时复用已启动进程，以快照引用为复用键，避免反复启停；
- **凭证最小权限**：按 credential refs 换短期凭证，session/沙箱结束后回收，不长期驻留明文；
- **装配失败 = attempt 失败**：按重试策略处理，编排层只收 attempt 失败信号，不感知装配细节；
- **边界**：编排层只透传引用，不解析内容；装配格式、MCP 启动机制、挂载路径由 Agent Core 实现，对编排层透明。

## 2. 工作流场景：run 级常驻沙箱

### 2.1 核心思想

**一个 run 一个沙箱，沙箱常驻整个 run；每个 step 是沙箱里的一个 session；所有 step 共享同一块文件工作区。**

```text
Sandbox（run 级常驻，文件共享）
  ├─ Session A（step1）  ── Attempt #1
  ├─ Session B（step2）  ── Attempt #1 → Attempt #2（续聊）
  └─ Session C（step3）  ── Attempt #1
       ▲
       └ 三个 session 共用同一块 workspace 文件区；step1 改的文件 step2/step3 直接可见
```

为什么这样设计：

- **文件天然共享，无需跨沙箱传递**：所有 step 同沙箱同盘，上游改动下游直接可见，不需要 git commit 接力或对象存储中转；
- **冷启动只付一次**：沙箱 boot（clone repo、装配环境）只在 run 开头一次（冷启 P95 < 60s），后续 step 落在热沙箱里；
- **与对话模型兼容**：进程可死、`--resume` 重载在常驻沙箱内发生，文件和对话本就都在，恢复更轻。

### 2.2 创建

- **触发时机**：run 内**第一个 step** 启动其首个 attempt 时；
- **创建与复用**：首个 step 触发创建沙箱，Agent Core 返回 `sandboxId`，Orchestration 记下 run ↔ sandboxId 映射；后续 step 的 attempt 带上该 `sandboxId`，命中同一沙箱；
- **创建动作**：分配运行环境 → 挂载 run 级 workspace 卷 → 按 `RepoRef` clone/checkout 代码仓 → 注入短期凭证（冷启 60s 来源）。

> 编排层视角：`StartAttempt` 带 `sandboxId`（首个 attempt 为空表示新建，Agent Core 在响应/事件中回报新建的 `sandboxId`；后续带已知 `sandboxId` 复用）。Orchestration 持有映射并负责复用判定，Agent Core 不感知 run/会话语义。

### 2.3 常驻（贯穿整个 run）

- **step 间切换**：上个 step 的 session 结束，下个 step 在同一沙箱开新 session，文件区不变；
- **空闲即冻结**：只要 run 未终止但当前没有活跃执行——沙箱空闲超阈值即冻结（释放算力、保留文件与对话上下文），需要时再解冻恢复。这覆盖三类空闲窗口：① step 挂起（`requiresConfirmation`）等用户确认/续聊；② attempt 失败后、重试退避等待期间；③ 任意 step 间的调度空档。冻结与聊天场景的 FROZEN 是**同一套机制**（见 §3.1），只是触发源不同（工作流由 run 内空闲触发，聊天由会话空闲触发）；解冻在确认/续聊/重试发起时进行；
- **进程退出 ≠ 沙箱销毁**：attempt 进程跑完退出后，沙箱与文件区、对话上下文都在，续聊在同沙箱内 `--resume`（沙箱已冻结则先解冻）。

续聊流程：

```text
step suspended（attempt #1 已产出结果）
   │ 用户带反馈点「继续对话」
   ▼
编排层创建 StepAttempt #2（带 resumeFromSessionRef）
   ├─ 进程仍存活（热）：直接在原进程续聊，低延迟
   └─ 进程已退出（冷）：在【同一沙箱内】起新进程，--resume <sessionRef> 重载后续聊
   ▼
attempt #2 succeeded ──► step 回到 suspended，等待下一次确认或续聊
```

### 2.4 销毁

run 进入终态时销毁：

| run 终态 | 沙箱处理 |
|---|---|
| `COMPLETED` | 销毁沙箱；workspace 卷与产物按数据策略保留/导出/清理 |
| `FAILED` | 销毁沙箱；workspace 与对话上下文清理归 Agent-Management workspace 生命周期管理 |
| `CANCELLED` | 销毁沙箱；对话上下文是否保留由取消请求指明，默认保留 |

- 销毁时回收运行时资源（容器、进程、临时凭证），**释放 workspace lease**；
- 编排层只释放 lease 引用，**不删卷**——卷的物理清理归 Agent-Management。

### 2.5 异常与超时

- **沙箱意外死亡**（节点故障/OOM）：in-flight attempt 失败，编排层按重试策略处理；原 `sandboxId` 失效，重试时新建沙箱（得到新 `sandboxId`）并 `--resume` 恢复对话，文件状态依赖 workspace 卷持久性；
- **run 级硬超时 / watchdog**：编排层判超时后取消 run，触发 §2.4 销毁。

### 2.6 step↔session 映射与文件持久性

- **一个 step = 一个 session**：step 启动首个 attempt 时新建 session 分配 `sessionRef`，attempt 结束回报，编排层持久化；
- **session 复用 = 续聊**：后续 attempt 带 `resumeFromSessionRef`，在原上下文上接着跑；
- **结构化结果传递走 StepOutput**：step 间除文件共享外，结构化结果（summary/result/artifactRefs）经 `StepOutput`，prompt 里用 `{{steps.xxx.result}}` 引用——与文件共享是两条互补通道；
- **workspace 持久性要求**：必须是沙箱内可跨进程持久的文件区，attempt 进程退出后改动对后续 step / 续聊仍可见——这是"文件共享"成立的前提；
- **并发**：MVP 串行单飞下，同一时刻 run 内只有一个 attempt 在写，无并发写冲突；续聊也按 conversation 串行。并行 DAG 留待后续。

## 3. 聊天场景：会话级惰性沙箱 + 快慢双路

聊天任务有三个特征，决定它**不能套用 workflow run 模型**：无预定义 DAG（一句话是一个意图）、连续对话（会追问改需求）、简繁混杂（「看下这函数」和「重构整个模块」走同一输入框）。硬把每句话包成单 step 的 AgentFlow 很别扭。所以聊天需要一条**独立的轻量执行路径**。

### 3.1 沙箱按会话绑定，惰性创建、四态分层

两条主线：

**(A) 沙箱按会话（conversation）绑定，不按"任务"。** 一个聊天窗口的多轮天然共享上下文和文件，所以一个会话对应一个沙箱、一个持续 session。复用 §2 的常驻思想——同样由 Orchestration 持有 conversation ↔ sandboxId 映射，首轮需执行时创建、后续轮凭 `sandboxId` 复用。

**(B) 沙箱惰性创建 + 四态分层。** 大规模下绝大多数轮次（纯问答）不碰文件、不调工具，**不该占用任何沙箱**。沙箱只在真正需要执行时才 provision，空闲后逐级回收。这是资源能否撑住规模的关键。

把"会话"与"沙箱"解耦成两个生命周期——会话是廉价持久对象（只存消息），沙箱是按需挂载的昂贵资源：

```text
会话（durable，廉价，只存消息历史 + sandbox 指针）
  └─ 沙箱挂载状态（昂贵，按需）：
       NONE ──首次需执行──► WARM ──空闲 idle-soft──► FROZEN ──空闲 idle-hard──► RECLAIMED
        ▲                     │                                                  │
        │                     └─────────── 用户新消息需执行 ◄────────────────────┘
        └─ 纯问答轮次：全程停在 NONE，不占沙箱
```

| 态 | 含义 | 占用 | 进入条件 | 主要服务目标 |
|---|---|---|---|---|
| **NONE** | 无沙箱，纯 LLM 对话 | 仅会话记录（KB 级） | 会话初始 / 当前轮无需执行 | 效率 |
| **WARM** | 沙箱存活，可立即执行 | 完整沙箱（容器+卷） | 某轮需碰文件/工具 | 体验（秒级响应）|
| **FROZEN** | 沙箱冻结，文件保留、不占算力 | 仅存储（卷快照） | WARM 后空闲超 idle-soft（建议 10min）| 效率 |
| **RECLAIMED** | 沙箱销毁，对话与文件外置持久 | 仅外置存储 | FROZEN 后空闲超 idle-hard（建议 1h），或会话关闭 | 效率 |

> 「主要服务目标」对应文档开头的三个设计目标：效率 = 规模化与资源效率、体验 = 用户体验流畅、安全 = 安全与隔离。

- **NONE→WARM**：某轮被判定需要执行（见 §3.3）时才惰性创建沙箱；
- **WARM→FROZEN**：冻结进程/容器但保留 workspace 卷快照，再来消息时解冻，比冷启快；
- **FROZEN→RECLAIMED**：彻底销毁，对话上下文与文件改动外置持久化（同 §1.2 的 sessionRef 落盘）；再来消息按冷启 + `--resume` 恢复；
- 用户感知的"聊天窗口"始终在；底下沙箱在各态流转对用户透明，只体现为响应延迟差异。

> 这套分层让"一万个开着的聊天窗口"的真实成本逼近"当前正在执行的少数会话"，而非一万个沙箱。**这是支撑大规模的核心。**

### 3.2 快慢双路

同一输入框，按任务量级分流：

**快路径（简单任务，默认，绝大多数）**：在会话沙箱里直接跑（NONE 则惰性升 WARM），就是多一轮对话。不开 run、不走编排、不进状态机。问答、读代码、改几行、跑个命令——都在会话沙箱里连续做，文件多轮共享。纯问答连沙箱都不要，停在 NONE。

**慢路径（复杂任务，少数，派生正式 run）**：当一句话其实是需长时间、多步骤、要审查的大活，**不在聊天沙箱里硬跑**，而是从聊天**派生一个工作流 run**——run 在自己独立的 run 级沙箱里跑重活（即 §2 那套），不阻塞聊天；聊天会话变成 run 的**遥控器 + 观察窗**（看进度、做确认、给反馈，事件流回聊天展示）。复杂任务由此自动获得 run 的全套能力：状态机、重试、审查 gate、可恢复、可观测。

### 3.3 怎么判断简单 vs 复杂：三层递进，不靠一次性预判

> **职责边界（两类判断分属两边）**：① 「是否需要沙箱」（NONE↔WARM）由 **Agent Core 自行判断**——等同于判断本轮 LLM 是否需要动用工具/碰文件，执行层天然掌握，无需编排层重复预判；② 「是否升级为 run」（快路径→派生工作流）由 **编排层 / 控制面决策**——派生 run、走状态机与审查 gate 是编排职责，Agent Core 不感知 run/DAG/审查，只**上报执行中的复杂度信号**（见 §5.3）供编排层判定。下文三层中，第一层的「升 WARM」属 ①，「提议建 run」与第二、三层均属 ②。

**核心原则：不要在用户发消息那一刻用一个分类器一次性判定"简单还是复杂"。** 同一句"修下登录 bug"，可能改一行就好，也可能牵出整个鉴权模块——**复杂度在执行前无法准确预知**。预判分类器要么过度保守（动不动建任务、体验差），要么漏判（大活塞进聊天沙箱跑到一半卡住）。

所以用三层递进，从"廉价预判"到"执行中暴露"：

**第一层 · 预判分流（快，只做粗筛）。** 发消息瞬间做一次轻量意图判断，**只筛两个极端**：
- 明显纯问答（"这段代码啥意思"）→ 停在 NONE，纯 LLM 回答，不要沙箱；
- 明显大工程（"把整个项目从 JS 迁到 TS"）→ 直接提议建 run；
- **中间地带（绝大多数）→ 一律先走快路径**，别纠结。

中间地带默认乐观走快路径，因为会话沙箱本就常驻，先试代价很低，把判断推迟到执行中。这层不需重模型，几个信号就够：动词（解释/问 vs 重构/迁移/批量）、是否提"整个/所有/全部"、是否涉多文件、是否要跑测试/构建/部署。规则 + 小模型兜底即可。

**第二层 · 执行中的复杂度护栏（真正可靠的判据）。** 核心——**让任务自己暴露复杂度，而非预测它**。快路径执行时设几条护栏，**任意一条触发就当场提议升级为 run**：

| 护栏信号 | 阈值（示例） | 为何是复杂度硬证据 |
|---|---|---|
| 工具调用/轮次数 | 本轮自主调用工具超 N 次（如 15）| 简单任务几步就完，停不下来说明是大活 |
| 单轮耗时 | 超 T 秒（如 60s）| 超出用户在聊天里能忍的等待 |
| 文件改动范围 | 改动文件超 M（如 10）或触及核心目录 | 大范围改动需审查，不该在聊天里悄悄发生 |
| 触发审查规则 | 命中危险操作/高风险路径（改鉴权、删数据、动 CI）| 必须进 run 的审查 gate |
| token/成本 | 单轮消耗超预算 | 成本失控信号 |

这些都是执行中才拿得到的真实信号，比预判准得多。触发时不硬切，而是给用户选择："这个改动比较大（已动 12 个文件），转成后台任务跑、跑完给你审查，可以吗？" 护栏的另一重作用是**兜底**：就算预判漏了大活，护栏也会在它跑失控前拦住，不拖垮会话沙箱——这是规模与体验的安全网。

**第三层 · 用户显式意图（最高优先级）。** 用户说了算，覆盖前两层：直接说"建个任务跑"→ 直接 run；界面给轻量开关主动标"大任务"；对升级提议说"不用就在这改"→ 尊重并留在快路径（但**审查规则那条护栏不可被覆盖**，安全底线）。

> 一句话：**预判只筛极端、中间地带乐观快跑、用执行中真实护栏定夺升级、用户意图永远优先。** 复杂度不是预测出来的，是任务跑起来后自己暴露的。

### 3.4 沙箱是否绑定 workspace/repo（惰性绑定）

- 会话可关联一个项目上下文（用户在哪个项目里发起聊天）。绑定项目 = 可访问该项目 repo 与连接器，能改代码；
- **repo 惰性 clone**：NONE→WARM 首次创建沙箱时才按需 clone，纯问答阶段不 clone；
- 无项目上下文的聊天 = 空沙箱，只支持问答与轻量文件操作，不碰 repo；
- "简单任务能干多重"取决于是否绑项目：绑了能在快路径直接改代码、跑测试；没绑就是纯助手。

### 3.5 派生 run 时的上下文交接

慢路径派生 run 时把聊天上下文带过去，否则 run 从零开始、用户得重述：

- **对话摘要注入**：把会话至今的关键意图、约束、已达成结论摘要后，作为派生 run 首个 step 的 prompt 上下文（或 `AgentFlow.variables`）；
- **文件状态交接**：若聊天沙箱已改 repo，run 需基于这些改动继续——会话沙箱先 commit、run 沙箱按该 commit checkout（**跨沙箱传递，用 git commit 作硬证据**，因聊天沙箱与 run 沙箱是两个沙箱、不共享卷）；
- **会话↔run 双向链接**：会话记录 `runId`，run 事件回流到会话展示；会话里的确认/反馈转成对 run 的 confirm/continue 调用。

### 3.6 Skill 与 MCP 加载

沿用 §1.3 的按需装配，触发点不同：NONE→WARM 创建沙箱时按会话关联的 `skillSnapshotRefs / mcpSnapshotRefs` 装配。会话存活期内能力集相对稳定（不像工作流每 step 切配置），MCP server 可长期复用，FROZEN 时随沙箱冻结、解冻时恢复。

## 4. 两场景对照与三目标落点

| 设计目标 | 工作流场景 | 聊天场景 |
|---|---|---|
| **规模化与资源效率** | run 数可控、生命周期明确；终态即销毁、释放 lease | 沙箱惰性创建（纯问答停 NONE）；四态分层让成本≈活跃会话数；WARM→FROZEN→RECLAIMED 逐级释放；冻结留卷快照避免重复 clone；复杂任务派生独立 run 不占聊天沙箱 |
| **用户体验流畅** | 状态机/重试/审查保障可靠交付 | 快路径秒级、对话连续、文件共享；沙箱态流转透明；复杂任务一键升级后台跑、带上下文不重述 |
| **安全与隔离** | run/租户沙箱互隔离；网络 egress 管控、工具 allowlist、危险操作进审查 gate；凭证短期最小权限 | 同左；空沙箱（无项目上下文）进一步收紧权限与出网 |

## 5. 对 Agent Core 的能力要求总结

本节是本文给 Agent Core 的接口/实现需求清单，按**优先级**分三档：

- **核心（P0）**：平台跑起来的最小集，缺则无法交付任何任务，MVP 第一天必须具备；
- **必要（P1）**：规模化与安全治理所需，缺则无法上量或合规，紧随核心落地；
- **非必要（P2）**：体验与效率增强，可后置，不阻塞主流程。

> **优先级前提**：① 当前里程碑 MVP 仅做工作流场景，聊天页延后到下一阶段，故**聊天专属能力归 P2**（沙箱冻结/解冻例外——工作流挂起也需要，已提至 P1），待聊天排期时整体提级。② **P0 只收纯功能最小集（执行 + 隔离 + 判活）**；资源限额、容量背压、冻结/解冻等「上规模才需打磨」的能力放 P1，资源指标、镜像治理等运营治理类放 P2——它们不阻塞功能交付。工作流场景的完整需求 = §5.1（P0）+ §5.2（P1）。

每项标注其类别（执行/隔离/资源/聊天）与适用场景。

### 5.1 核心（P0）——不具备则无法交付

| 能力 | 类别 | 说明 |
|---|---|---|
| 沙箱创建 + 按 ID 复用 | 执行 | 创建沙箱并返回 `sandboxId`；后续凭 `sandboxId` 复用同一沙箱。Agent Core 无需理解 run/会话语义，run/会话 ↔ sandboxId 的映射由 Orchestration 持有 |
| 沙箱常驻与多 session | 执行 | 单沙箱承载多 session，session 间文件共享、对话上下文隔离 |
| 进程可死 + 对话恢复 | 执行 | 对话上下文落盘（键=sessionRef），进程退出不丢，`--resume` 重载续聊 |
| 按需装配 skill/MCP/知识库 | 执行 | 按引用拉取装配；凭证最小权限注入、用后回收 |
| workspace 挂载与 repo clone | 执行 | 挂载持久卷；按 `RepoRef` clone/checkout 代码仓 |
| 事件回报 | 执行 | 生命周期事件 + Agent 过程事件（thinking/message/tool_*/std*），统一协议上报 |
| 租户 / 任务隔离 | 隔离 | 不同 run/会话/租户沙箱互隔离，文件系统、进程、网络命名空间互不可见 |
| 凭证隔离与短期化 | 隔离 | 短期令牌、最小权限、作用域限本沙箱、用后回收；不持久化明文、不回显凭证值 |
| 文件系统边界 | 隔离 | Agent 只能访问策略允许的挂载路径，越界访问被拒 |
| 运行状态查询 | 执行 | 提供按 attempt/沙箱查询运行状态的接口，供编排层轮询判活、配合超时与 watchdog 回收（无需主动推 heartbeat）|

> P0 是纯功能最小集（执行 + 隔离 + 判活）。资源限额、容量、性能、可观测、治理等「上规模才需要打磨」的能力均下沉到 P1/P2。

### 5.2 必要（P1）——规模化与安全治理所需

| 能力 | 类别 | 说明 |
|---|---|---|
| 网络策略（egress 管控） | 隔离 | 默认最小出网，仅放行 allowlist（模型端点/已批准连接器/repo 源）；空沙箱进一步收紧 |
| 工具 / 命令 allowlist | 隔离 | 可执行工具、命令、路径受策略约束，不开放任意脚本与网络访问 |
| 危险操作拦截 | 隔离 | 命中高风险操作（改鉴权/删数据/动 CI/越权）按审查策略阻断或上报，作为护栏与审查 gate 的执行底座 |
| 资源限额 | 资源 | 沙箱可施加资源配额并在超限时按策略处理，避免单沙箱影响同节点其他沙箱（具体配额维度与策略由 Agent Core 定）|
| 容量与背压 | 资源 | 容量不足时 `StartAttempt` 以可识别状态快速拒绝，不长时间阻塞；展示类事件可限流，但不阻塞控制类事件（attempt.result/终态/状态查询）|
| 沙箱冻结 / 解冻 | 资源 | **两场景共用**：沙箱空闲超阈值时冻结（释放算力、保留文件与对话上下文），需要时解冻恢复。工作流由 run 内空闲触发（挂起等确认、重试退避、step 间空档），聊天由会话空闲触发，避免空闲沙箱长期空占算力（实现形态由 Agent Core 定）|

> P1 为上规模与安全合规所需的隔离 + 资源治理能力，不含聊天专属逻辑。聊天专属能力见下方 P2。

### 5.3 非必要（P2）——聊天能力 + 资源/效率优化，可后置

聊天页延后（整组聊天能力）+ 上规模才需打磨的资源治理与效率优化，归此档，不阻塞 MVP 工作流主流程：

| 能力 | 类别 | 说明 |
|---|---|---|
| 无沙箱对话执行 | 聊天 | 纯问答轮次只做 LLM 推理、不创建沙箱、流式返回——大规模省资源的前提 |
| 轮次级意图预判 | 聊天 | 轻量把轮次粗分为「纯问答/需执行/明显大活」，决定是否升 WARM 或提议派生 run |
| 无状态低延迟（非沙箱轮次） | 聊天 | 非沙箱轮次可水平扩展、无沙箱绑定，支撑高并发问答 |
| 四态自动流转编排 | 聊天 | 在 NONE/WARM/FROZEN/RECLAIMED 间按空闲时长自动流转（NONE↔WARM 惰性升降、RECLAIMED 外置持久化）；冻结/解冻基础能力见 P1 |
| 执行中复杂度信号上报 | 聊天 | 实时暴露工具调用数/耗时/改动范围/命中规则，供护栏判定升级（无则护栏用粗略信号兜底）|
| 跨沙箱状态交接 | 聊天 | 聊天沙箱 commit、run 沙箱按 commit checkout，承接慢路径派生 |
| 资源指标可观测 | 资源 | 上报 token/成本/耗时/资源占用等指标，供成本归属、配额、告警 |
| 镜像与运行时治理 | 资源 | 执行环境来源可治理、可复现、可审计 |
| MCP server 复用 | 执行 | 相邻 session 同引用复用已启动 MCP server，避免反复启停 |

> 当聊天页排期确定时，聊天类各项应整体提级（无沙箱执行/预判/四态流转属聊天的 P0~P1）；资源指标、镜像治理、MCP 复用为长期优化，可继续保持 P2。

### 5.4 编排层（Agent-Orchestration）边界

- **工作流场景**：全程经编排层；StartAttempt 契约为 `workspaceRef + resumeFromSessionRef + sandboxId`（`sandboxId` 首个 attempt 为空、由 Agent Core 回报后 Orchestration 持有并在后续 attempt 回传）；
- **聊天快路径**：建议**直连 Agent Core 会话执行接口，不经编排层**，避免轻量交互被编排开销拖累；
- **聊天慢路径**：派生 run 后回到编排层标准流程。

## 变更摘要

- 本次变更：新增团队空间 RBAC 权限管理功能设计，定义创建者、管理员、团队成员和访客四类基础角色，并补充租户管理员不默认拥有团队空间内容写权限的边界。
- 影响范围：产品定义、功能索引、租户上下文、团队空间访问鉴权、成员与角色管理、Harness 配置权限、页面按钮权限态、后端接口鉴权和验收标准。
- 待确认问题：
  - devuc 的鉴权接口、角色同步字段、失败错误码和降级策略。
  - 创建者是否支持后续移交或退出团队空间；第一版暂按不支持移交和退出处理。
  - 团队成员除 Harness 配置外是否允许更多空间内容管理操作；第一版暂按仅查看和使用处理。
  - 租户管理员是否需要团队空间级只读穿透能力；第一版暂按仅可查看团队空间清单，不默认进入空间内容处理。

## PRD 设计

### 用户目标

- 团队创建者可以创建团队空间，并默认拥有管理员权限。
- 团队创建者和管理员可以管理空间成员与角色，并可以额外添加管理员。
- 租户管理员可以治理本租户团队空间清单，但不因租户管理员身份自动获得具体团队空间的内容写权限。
- 团队成员可以进入被授权团队空间，查看空间内容并使用已配置的 Agent 能力，但不能配置 Harness。
- 访客可以查看团队空间所有内容，但不能执行新增、编辑、删除、配置、邀请、运行等操作。
- 平台可以对接公司 devuc 服务完成身份与权限鉴权，保证空间访问和操作边界一致。

### 范围

**包含：**

- 团队空间基于 RBAC 的基础权限管理。
- 团队空间归属租户后的权限上下文校验。
- 对接公司 devuc 服务进行空间访问和操作鉴权。
- 创建者、管理员、团队成员、访客四类基础角色。
- 创建者默认具备管理员权限。
- 创建者和管理员可添加管理员，也可添加团队成员和访客。
- 访客可查看团队空间所有内容，但没有任何操作权。
- 团队成员可查看团队空间内容并使用已配置能力，但没有 Harness 配置权限。
- 前端根据角色展示可用操作、禁用操作或只读状态。
- 后端对所有写操作和 Harness 配置操作进行权限校验。
- 租户管理员未被授予团队空间角色时，仅具备租户管理面能力，不具备团队空间内容写权限。

**不包含：**

- 复杂权限矩阵配置后台。
- 自定义角色和自定义权限点。
- 角色审批流。
- 跨团队、跨空间权限继承。
- 租户管理员自动继承团队空间管理员权限。
- 字段级、文件级或资源实例级精细权限。
- devuc 服务自身的账号、组织、权限策略和主数据治理。
- 创建者移交、空间所有权变更和历史权限迁移。

### 核心流程

1. 用户访问团队空间时，系统获取用户身份、目标租户和目标团队空间，并对接 devuc 校验用户是否具备访问该租户和空间的权限。
2. 鉴权通过后，系统查询用户在当前团队空间内的角色：创建者、管理员、团队成员或访客。
3. 系统根据租户上下文和团队空间角色映射当前用户可查看内容、可使用能力和可执行操作。
4. 前端根据权限结果渲染页面：有权限的操作正常展示，无权限的写操作隐藏、禁用或展示只读状态。
5. 用户在租户下创建团队空间时，系统将创建人记录为创建者，并默认赋予管理员权限。
6. 创建者或管理员进入成员管理入口，搜索并添加用户，选择管理员、团队成员或访客角色。
7. 创建者或管理员保存成员角色变更后，后端再次校验操作者权限，并写入空间成员关系。
8. 团队成员访问 Harness 配置入口时，系统允许查看必要信息，但禁止新增、编辑、删除和保存配置。
9. 访客访问团队空间时，系统允许查看空间所有内容，但不允许触发任何操作。
10. 无权限用户调用受限接口时，后端拒绝请求，前端展示权限不足提示。

### 业务规则

- 团队空间必须归属一个租户，团队空间 RBAC 判断必须带有租户上下文。
- 创建者是团队空间的创建人，默认具备管理员权限。
- 第一版创建者不支持被移除、降级或移交。
- 管理员具备团队空间管理权限，包括成员管理、角色分配、知识库配置和 Harness 配置。
- 创建者和管理员可以添加新的管理员。
- 团队成员可以查看团队空间内容并使用已配置的 Agent 能力，但不能进行 Harness 配置。
- 访客拥有全空间只读访问权，不具备任何操作权。
- Harness 配置操作包括新增、编辑、删除、启用、停用、保存、回写和从公司 AI 市场选择配置资源。
- 写操作必须以后端鉴权结果为准，前端权限态只作为交互体验优化。
- 同一用户在同一团队空间第一版仅保留一个角色。
- 同一用户可同时是租户管理员和某个团队空间成员；团队空间内操作权限以团队空间角色为准。
- 租户管理员可以在租户管理面查看本租户团队空间清单，但不默认具备进入团队空间内容、编辑成员、配置知识库或配置 Harness 的权限。
- 若用户未加入团队空间且 devuc 未授予访问权限，不能访问该团队空间。
- devuc 鉴权失败、超时或返回不可信结果时，系统应默认拒绝写操作。
- 所有成员和角色变更需要记录审计信息，至少包含操作者、目标用户、变更前角色、变更后角色、时间和结果。

### 权限矩阵

| 权限项 | 创建者 | 管理员 | 团队成员 | 访客 | 租户管理员（未授予团队角色） |
| --- | --- | --- | --- | --- | --- |
| 访问团队空间 | 是 | 是 | 是 | 是 | 否 |
| 查看团队空间所有内容 | 是 | 是 | 是 | 是 | 否 |
| 查看租户团队空间清单 | 否 | 否 | 否 | 否 | 是 |
| 使用已配置 Agent 能力 | 是 | 是 | 是 | 否 | 否 |
| 管理成员与角色 | 是 | 是 | 否 | 否 | 否 |
| 添加管理员 | 是 | 是 | 否 | 否 | 否 |
| 管理知识库配置 | 是 | 是 | 否 | 否 | 否 |
| 配置 Harness | 是 | 是 | 否 | 否 | 否 |
| 触发写操作 | 是 | 是 | 否 | 否 | 否 |

### 边界与异常

- 用户没有团队空间访问权限时，系统拒绝进入空间，并提示无访问权限。
- devuc 鉴权失败或超时时，系统应提示稍后重试，并禁止进入或禁止执行受限操作。
- 用户角色为空、角色被删除或角色数据不一致时，系统按无权限处理。
- 团队成员尝试保存 Harness 配置时，系统拒绝提交并提示无配置权限。
- 访客尝试执行任意操作时，系统拒绝提交并提示访客仅可查看。
- 管理员添加管理员失败时，系统保留已填写信息，并提示失败原因。
- 成员角色变更后，用户已打开页面的权限状态需要在下次刷新、重新进入页面或接口返回权限不足时更新。
- 团队空间至少需要保留创建者；第一版不支持删除创建者。
- 租户管理员尝试以租户身份进入未授权团队空间内容时，系统应提示需要团队空间授权。

## UI 设计

### 是否需要刷新

- 结论：是
- 理由：需要新增或调整成员与角色管理入口、角色选择控件、只读状态、权限不足提示和 Harness 配置按钮权限态。

### 页面与交互

- 成员管理页展示成员姓名、账号、角色、加入状态和操作入口。
- 页面需要展示当前团队空间所属租户，便于用户确认权限上下文。
- 创建者和管理员可以添加成员、修改角色、移除非创建者成员。
- 角色选择第一版仅包含管理员、团队成员和访客；创建者由系统自动标识。
- 团队成员进入 Harness 配置相关页面时，不展示保存、删除、市场选择和回写类操作。
- 访客进入团队空间时，页面整体保持只读，不展示任何操作类主按钮。

### 状态与文案

- 空状态：暂无成员，创建者可添加成员。
- 成功态：成员已添加、角色已更新、管理员已添加。
- 异常态：无访问权限、权限不足、devuc 鉴权失败、成员添加失败、角色更新失败。
- 只读态：当前角色仅支持查看。
- 租户管理员无团队角色态：仅可返回租户管理面查看团队空间清单。

## 前端设计

### 是否需要刷新

- 结论：是
- 理由：前端需要根据后端返回的角色和权限渲染页面状态，并在受限操作上提供一致的拦截与提示。

### 页面、组件与状态

- 页面路由：
  - 团队空间成员管理页。
  - 团队空间权限设置页或成员管理中的角色配置区域。
  - 租户管理面团队空间清单入口的权限跳转状态。
- 核心组件：
  - MemberList：成员列表。
  - MemberRoleSelect：角色选择。
  - AddMemberDialog：添加成员弹窗。
  - PermissionGuard：基于权限控制按钮、菜单和表单状态。
  - TenantScopeBadge：展示当前空间所属租户和权限来源。
  - ReadonlyBanner：只读角色提示。
- 状态管理：
  - 当前用户角色。
  - 当前租户上下文和团队空间归属。
  - 当前用户权限集合。
  - 成员列表与角色变更提交状态。
  - devuc 鉴权状态和权限不足状态。

### 接口依赖与异常处理

- 依赖接口：
  - 查询当前用户在团队空间内的角色和权限。
  - 查询当前团队空间所属租户和用户租户级权限。
  - 查询团队空间成员列表。
  - 搜索可添加用户。
  - 添加成员并授予角色。
  - 更新成员角色。
  - 移除成员。
- 异常处理：
  - 接口返回权限不足时，前端切换到只读态或展示权限不足提示。
  - 成员添加或角色更新失败时，保留用户输入并提示原因。
  - devuc 鉴权失败时，禁止继续提交受限操作。

## 后端设计

### 是否需要刷新

- 结论：是
- 理由：后端需要接入 devuc 鉴权，维护空间成员角色关系，并对成员管理、知识库管理和 Harness 配置写接口执行统一权限校验。

### 接口与数据

- 核心服务：
  - PermissionService：封装 RBAC 权限判断。
  - DevucAuthAdapter：对接公司 devuc 服务完成身份和访问鉴权。
  - SpaceMembershipService：管理空间成员、角色和成员状态。
  - TenantScopePermissionResolver：合并租户上下文和团队空间 RBAC 判断。
  - AuditService：记录权限相关操作审计。
- 核心数据对象：
  - SpaceMember：租户 ID、空间 ID、用户 ID、角色、加入状态、创建时间、更新时间。
  - RolePermission：角色、权限项、是否允许。
  - PermissionAuditLog：空间 ID、操作者、目标用户、操作类型、变更前后内容、结果、时间。

### 业务规则、权限与异常处理

- 每次访问团队空间前需完成 devuc 鉴权。
- 每次团队空间鉴权必须校验空间所属租户与请求上下文一致。
- 每次执行写操作前需完成后端 RBAC 权限校验。
- 角色权限第一版采用系统内置映射，不提供后台自定义。
- 创建者角色由空间创建动作生成，不通过成员管理手动创建。
- 创建者和管理员可以新增管理员、团队成员和访客。
- 租户管理员不默认继承团队空间管理员权限；若其被添加为团队空间管理员，则按团队空间管理员权限执行。
- 团队成员不允许调用 Harness 配置写接口。
- 访客不允许调用任何写接口或运行类接口。
- 鉴权失败、角色缺失或权限映射异常时，默认拒绝操作。
- 权限变更和拒绝操作需要记录审计日志，便于后续追踪。

## 验收标准

### 正常路径

- 用户创建团队空间后，系统将该用户标记为创建者，并默认拥有管理员权限。
- 创建者可以添加管理员、团队成员和访客。
- 管理员可以添加管理员、团队成员和访客。
- 团队成员可以查看团队空间内容，并使用已配置的 Agent 能力。
- 访客可以查看团队空间所有内容。
- 后端可以通过 devuc 完成空间访问鉴权。
- 租户管理员可以查看本租户团队空间清单，但在未被授予团队角色时不能进入团队空间内容管理。

### 边界场景

- 同一用户在同一团队空间仅拥有一个角色。
- 同一用户同时具备租户管理员和团队空间成员身份时，团队空间内操作按团队空间角色生效。
- 创建者在成员列表中展示为创建者，并同时具备管理员能力。
- 第一版不允许删除、降级或移交创建者。
- 角色变更后，用户重新进入空间时按最新角色生效。

### 异常场景

- devuc 鉴权失败时，系统拒绝访问或拒绝执行受限操作。
- 未加入团队空间且未被 devuc 授权的用户不能访问空间。
- 租户管理员未被授予团队角色时，尝试进入团队空间内容管理会被拒绝。
- 成员添加失败时，系统提示失败原因，并允许创建者或管理员重试。
- 角色更新失败时，系统不应产生部分成功的脏数据。

### 权限场景

- 团队成员不能新增、编辑、删除、保存或回写 Harness 配置。
- 访客不能新增、编辑、删除、保存、回写、邀请成员或触发运行类操作。
- 团队成员和访客调用写接口时，后端返回权限不足。
- 租户管理员未被授予团队空间管理员角色时，不能编辑该团队空间成员、知识库或 Harness 配置。
- 创建者和管理员的成员角色变更会写入审计日志。

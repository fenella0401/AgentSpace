## PRD 设计

### 用户目标

- 对话创建人可以看到 Agent 生成或修改的文档变更提案。
- 具备提案内容查看权限的用户可以查看文档内容和 diff，判断 Agent 建议变更的影响。
- 对话创建人可以接受变更，将提案写入正式知识库文档。
- 对话创建人可以拒绝变更，保持正式文档为修改前版本。
- 对话创建人在接受变更时可以得到版本冲突提醒，避免覆盖他人的更新。
- 用户可以识别文档变更提案的处理状态，避免重复处理。

### 业务规则

- 文档变更必须归属一个 Agent 会话，并继承会话的租户和团队空间上下文。
- Agent 文档变更事件必须先形成待接受 `DocumentChangeProposal`，不得直接覆盖正式知识库文档。
- 文档变更提案展示文档名称、路径、变更类型和提案状态。
- 提案状态包括待接受、已接受、已拒绝、冲突。
- 支持查看 proposed version 内容。
- 生成新文档提案的 base version 可以为空，diff 以空文档与 proposed version 对比。
- 修改已有文档提案必须记录 base version，diff 以 base version 与 proposed version 对比。
- 完整内容和 diff 查看必须同时满足团队空间访问权限、会话查看权限和提案内容查看权限。
- 修改已有文档的提案内容查看权限按对应知识库文档查看权限判断。
- 生成新文档的提案内容查看权限按目标路径所在知识库目录查看权限判断。
- 首版只有会话创建人可以接受或拒绝文档变更。
- 其他具备查看权限的用户和访客只读查看文档变更，不展示接受或拒绝操作。
- 接受生成新文档提案前必须校验目标路径可写入且目标路径不存在；目标路径已存在时阻止接受，将提案状态更新为冲突，并提示路径冲突。
- 接受生成新文档提案成功后创建正式知识库文档。
- 接受修改已有文档提案前必须校验正式文档仍处于提案 base version。
- 修改已有文档时 base version 不一致必须阻止写入正式文档，并将提案状态更新为冲突。
- 接受成功后写入正式知识库文档，提案状态变为已接受。
- 拒绝后不写入正式文档，提案状态变为已拒绝。
- 已接受、已拒绝或冲突的提案不允许重复处理。
- 接受、拒绝和冲突都需要记录审计信息。
- F-007 不提供 Agent 会话创建、运行、询问回答、会话看板和详情时间线能力。

### 边界与异常

- agent core 文档变更事件缺少必要字段时，系统不创建可处理提案，并记录会话事件错误。
- 文档提案内容加载失败时，系统保留提案卡片并允许重试。
- diff 生成或加载失败时，系统展示失败提示，不影响用户查看 proposed version 内容。
- 提案接受时正式文档已被修改，系统阻止写入，将提案状态更新为冲突，并提示版本冲突。
- 提案接受时写入知识库失败，系统保持正式文档不变，提案保持待接受，并提示用户重试。
- 非会话创建人尝试接受或拒绝时，后端返回权限不足。
- 无会话查看权限或提案内容查看权限的用户不能查看文档变更提案内容和 diff。
- 已处理提案再次提交接受或拒绝时，系统返回状态已变化。
- 生成新文档提案接受时目标路径已存在，系统阻止接受，将提案状态更新为冲突，并提示路径冲突。
- 修改已有文档提案接受时，正式文档已不存在或无法读取当前版本，系统阻止接受并提示处理失败。

## 验收标准

### 正常路径

- agent core 返回文档变更事件后，系统创建待接受文档变更提案。
- agent core 返回生成新文档事件时，系统允许 base version 为空，并以空文档与 proposed version 生成 diff。
- agent core 返回修改已有文档事件时，系统记录 base version，并以 base version 与 proposed version 生成 diff。
- 文档变更卡片展示文档名称、路径、变更类型和待接受状态。
- 具备查看权限的用户可以查看 proposed version 内容。
- 具备查看权限的用户可以查看 base version 与 proposed version 的 diff。
- 会话创建人接受生成新文档提案后，系统创建正式知识库文档，提案状态变为已接受。
- 会话创建人接受修改已有文档提案后，正式文档更新为 proposed version，提案状态变为已接受。
- 会话创建人拒绝文档变更后，正式文档保持 base version，提案状态变为已拒绝。
- 文档变更接受、拒绝和冲突均写入审计日志。

### 边界场景

- diff 加载失败时，不影响查看文档提案摘要和 proposed version 内容。
- 已接受、已拒绝或冲突提案不再展示接受或拒绝按钮。
- 非会话创建人可只读查看文档变更摘要。
- 生成新文档目标路径已存在时，系统阻止接受，将提案状态更新为冲突，并提示路径冲突。
- 修改已有文档时，正式文档已不存在或无法读取当前版本，系统阻止接受并提示处理失败。
- agent core 文档变更事件缺少必要字段时，系统不创建可处理提案，并记录会话事件错误。

### 异常场景

- 接受时发现正式文档版本与 base version 不一致，系统阻止写入，将提案状态更新为冲突，并提示版本冲突。
- 接受时写入知识库失败，正式文档保持不变，提案保持待接受，并提示用户重试。
- 文档内容加载失败时，系统展示失败提示并允许重试。
- 已处理提案再次提交处理时，系统返回状态已变化。

### 权限场景

- 非会话创建人不能接受或拒绝文档变更。
- 访客不能接受或拒绝文档变更。
- 无会话查看权限的用户不能查看文档变更提案。
- 无提案内容查看权限的用户不能查看文档变更内容和 diff。

## UI 设计

### 是否需要刷新

- 结论：是
- 理由：需要新增文档变更卡片、内容查看、diff 查看、接受、拒绝、冲突和已处理状态。

### 页面与交互

- 文档变更卡片：
  - 展示文档名称、路径、变更类型、提案状态和更新时间。
  - 提供查看内容和查看 diff 入口。
- 文档内容查看：
  - 展示 proposed version 内容。
  - 标识文档路径和变更类型。
- diff 查看：
  - 展示 base version 与 proposed version 的差异。
  - diff 加载失败时展示错误和重试入口。
- 提案处理：
  - 待接受状态下，会话创建人可点击接受或拒绝。
  - 已接受、已拒绝或冲突状态下，不展示提交类按钮。
  - 冲突状态展示正式文档已变化，需要用户重新查看或等待后续处理。

### 状态与文案

- 加载态：文档内容加载中、diff 加载中、接受中、拒绝中。
- 等待态：文档变更等待确认。
- 成功态：文档变更已接受、文档变更已拒绝。
- 异常态：内容加载失败、diff 加载失败、文档版本冲突、权限不足、提案状态已变化。
- 只读态：当前用户仅可查看，不能接受或拒绝。

## 前端设计

### 是否需要刷新

- 结论：是
- 理由：需要新增文档变更卡片、diff 查看器、内容查看、接受/拒绝操作和冲突处理。

### 页面、组件与状态

- 核心组件：
  - DocumentChangeCard：展示文档变更提案摘要。
  - DocumentContentViewer：展示 proposed version 内容。
  - DocumentDiffViewer：展示 base 与 proposed 的 diff。
  - ProposalActionBar：接受、拒绝和状态提示。
  - ProposalConflictBanner：展示版本冲突。
- 状态管理：
  - 当前 Agent 会话和文档提案上下文。
  - 文档内容加载状态。
  - diff 加载状态。
  - 接受、拒绝提交状态。
  - 提案状态和冲突状态。

### 接口依赖与异常处理

- 依赖接口：
  - 查询文档变更提案详情。
  - 查询文档变更提案内容。
  - 查询文档变更 diff。
  - 接受文档变更提案。
  - 拒绝文档变更提案。
- 异常处理：
  - 内容加载失败时允许重试。
  - diff 加载失败时允许重试，不阻塞返回详情页。
  - 接受发生版本冲突时刷新提案状态并展示冲突提示。
  - 接口返回权限不足时切换为只读态或展示权限不足提示。

## 后端设计

### 是否需要刷新

- 结论：是
- 理由：需要新增文档变更提案、diff 引用、知识库写入、版本校验、接受、拒绝和审计能力。

### 接口与数据

- 核心服务：
  - DocumentChangeProposalService：管理文档变更提案、内容、diff、接受和拒绝。
  - KnowledgeDocumentService：读取正式文档版本并写入 accepted proposed version。
  - DocumentDiffService：生成或读取 base 与 proposed 的 diff。
  - PermissionService：判断文档变更查看和处理权限。
  - AuditService：记录接受、拒绝和冲突。
- 核心数据对象：
  - DocumentChangeProposal：提案 ID、会话 ID、事件 ID、文档 ID 或路径、变更类型、base version、proposed version、diff 引用、状态、创建时间、更新时间。
  - DocumentChangeProposalStatus：待接受、已接受、已拒绝、冲突。
  - DocumentChangeType：生成、修改。
- 建议接口：
  - `GET /team-spaces/{spaceId}/document-change-proposals/{proposalId}`
  - `GET /team-spaces/{spaceId}/document-change-proposals/{proposalId}/content`
  - `GET /team-spaces/{spaceId}/document-change-proposals/{proposalId}/diff`
  - `POST /team-spaces/{spaceId}/document-change-proposals/{proposalId}/accept`
  - `POST /team-spaces/{spaceId}/document-change-proposals/{proposalId}/reject`

### 业务规则、权限与异常处理

- 查询提案详情时必须先校验团队空间访问权限、会话查看权限和提案内容查看权限。
- 接受或拒绝提案时必须校验操作者是会话创建人。
- 接受生成新文档提案前必须校验目标路径可写入且目标路径不存在。
- 接受修改已有文档提案前必须校验正式文档当前版本等于提案 base version。
- 冲突时不得写入正式文档。
- 生成新文档目标路径已存在时，提案状态更新为冲突。
- 拒绝提案不得修改正式文档。
- 已接受、已拒绝或冲突的提案不允许重复处理。
- 接受、拒绝和冲突均需写入审计日志。

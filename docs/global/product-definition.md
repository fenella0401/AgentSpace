## 3. 产品定位

AgentSpace 是面向公司内部团队的 Agent 工作空间产品。

团队可以在 AgentSpace 中创建团队空间，配置任务触发规则和 Agent 协作方式；系统根据事件、定时计划或人工指令，自动触发单 Agent（在前端界面统一简称 Agent） 或 AgentFlow 执行。

人在 Agent 执行过程中主要承担监管、审查、审批和干预职责，在关键节点实现 human in the loop。系统通过看板、执行记录、文件变更和审计链路，帮助团队追踪 Agent 工作过程和最终交付结果。

一句话定位：

> AgentSpace 是团队级 Agent 工作空间，用于配置、触发、监管和追踪 Agent 协同完成团队任务的全过程。

AgentSpace 首先以需求开发、架构设计看护、代码变更、测试集成、发布状态追踪等软件生产场景打样穿刺。需考虑可扩展性，长期可扩展到其他 Agent 协作的团队任务场景。

## 4. 产品目标和价值

AgentSpace 的核心目标是：

1. 让团队可以围绕具体项目或业务场景创建独立 Agent 工作空间。
2. 让团队管理员可以配置任务触发方式和 Agent 协作方式。
3. 让团队成员可以通过事件任务、定时任务或临时任务触发 Agent 工作。
4. 让 Agent 可以基于团队空间内授权的 Skill、Tool、知识库和环境变量执行任务。
5. 让复杂任务可以通过 AgentFlow 编排多个 Agent 角色工作步骤进行长程处理。
6. 让人在关键节点查看、审批、驳回、补充信息或修改指令。
7. 让团队可以通过看板查看任务执行进度、Agent 运行记录、AgentFlow 执行链路和 Agent 执行结果与产出。
8. 让所有 Agent 执行过程具备可追踪、可审计、可复盘能力。

AgentSpace 的核心不是一个普通聊天产品，也不是一个传统项目管理系统，而是一个团队级 Agent 工作空间。

AgentSpace 的关键价值是：

1. 将团队任务配置化。
2. 将 Agent 能力工程化。
3. 将 Agent 执行过程可视化。
4. 将人的审查和干预机制产品化。
5. 将 Agent 产出和文件变更沉淀为可追踪交付件。
6. 将团队与 Agent 的协作过程变成可治理、可审计、可复盘的工作流。

AgentSpace 最终希望帮助团队从“临时使用 Agent”升级为“配置和运营 Agent 工作系统”。

## 5. 目标用户与角色

### 5.1 系统管理员

系统管理员负责平台级治理能力，包括：

* 创建租户。
* 授予租户管理员。
* 注册平台级事件源类型和开放范围。
* 管理平台级 AgentRuntime、模型、Skill、Tool、知识库模板等资源。

### 5.2 租户管理员

租户管理员负责本租户内的治理能力，包括：

* 管理租户内团队空间清单。
* 注册租户级事件源类型和开放范围。
* 配置租户级 AgentRuntime、模型、Skill、Tool、知识库模板等资源。

### 5.3 团队创建者

团队创建者负责创建团队空间。

默认情况下，团队创建者同时拥有团队管理员和团队成员权限。

### 5.4 团队管理员

团队管理员负责团队空间内的配置，包括：

* 管理团队成员。
* 配置团队空间 Harness 执行工程，包括 Skill、Tool、环境变量、知识库、Agent角色、AgentFlow。
* 配置团队空间任务和执行方式。
* 审查团队空间内的重要执行结果和文件变更。

### 5.5 团队成员

团队成员是 AgentSpace 的主要使用者，包括：

* 发起临时任务（作为补充入口，不作为 AgentSpace 的主任务组织方式）。
* 与 Agent 协同完成任务，在被授权的节点审批、驳回或补充信息。
* 查看团队空间看板。

默认情况下，团队创建者和团队管理员同时也是团队成员。

### 5.6 访客

访客只能查看被授权的团队空间资源，不可修改任何资源。

访客可以查看：

* 团队空间基础信息。
* 看板。
* 已授权的执行记录。
* 已授权的知识库内容。

## 6. 核心对象模型

AgentSpace 的核心对象分为配置态对象和运行态对象。

### 6.1 配置态对象

配置态对象用于描述团队空间中“可以怎么执行”。

主要包括：

```text
Tenant 租户
  └── Space 团队空间
        ├── Member 成员与权限
        ├── Harness 执行工程配置
        │     ├── Capability Center 能力中心
        │     │     ├── Skill
        │     │     ├── Tool
        │     │── Knowledge 知识库（Space Agent.md）
        │     ├── Environment Variables 环境变量
        │     ├── Agent Role Agent 角色
        │     └── AgentFlow Agent 协作流
        └── Task Config 任务配置
              ├── Event Task Config 事件任务配置
              └── Scheduled Task Config 定时任务配置
```

### 6.2 运行态对象

运行态对象用于描述团队空间中“某一次实际执行”。

主要包括：

```text
Task Run
  ├── source_type: event / scheduled / temporary
  ├── Agent Run
  └── AgentFlow Run
```

### 6.3 配置态和运行态的关系

| 配置态对象          | 运行态对象              | 说明                                |
| -------------- | ------------------ | --------------------------------- |
| 事件任务配置         | 事件任务执行记录           | 事件发生后生成一次执行记录                     |
| 定时任务配置         | 定时任务执行记录           | 到达调度时间后生成一次执行记录                   |
| Agent 角色       | Agent Run          | Agent Run 启动时读取对应 Agent 角色配置      |
| AgentFlow      | AgentFlow Run      | AgentFlow Run 是 AgentFlow 的一次运行实例 |
| AgentFlow Step | AgentFlow Step Run | Step Run 是某个步骤的一次实际执行             |
| Harness        | 执行上下文              | Agent 执行时读取 Harness 中的能力、知识、变量和规则 |

## 8. 关键概念详解

### 团队空间

团队空间是 AgentSpace 的基本工作单元。

一个团队空间通常对应一个团队、项目、业务场景或长期任务域，例如：

* AgentSpace 项目团队空间。
* AgentCore 项目团队空间。

团队空间内包含：

1. 成员与权限。
2. Harness 配置。
3. 任务配置。
4. Agent 执行记录。
5. AgentFlow 执行记录。
6. 看板。
7. 文件变更记录。

团队空间可以是多人协作空间，也可以是单人工作空间。

---

### Harness 配置

Harness 是团队空间内用于约束和驱动 Agent 执行的一组工程化配置。

它描述某类任务应该由哪些 Agent、使用哪些 Skill、Tool、知识库和环境变量，按照什么流程执行，在什么节点需要人工介入，以及如何判断任务完成。

一句话定义：

> Harness = Agent 执行工程配置。

Harness 由以下部分组成：

1. 能力中心。
2. 知识库。
3. 环境变量。
4. Agent 角色。
5. AgentFlow。

当前阶段，一个团队空间默认拥有一个全局 Harness。未来可扩展为一个团队空间支持多个 Harness，用于不同任务类型。

每个部分可以编辑和保存，Harness 全局有一个 Publish 功能，选择后把配置推送到 AgentCore 对应团队空间租户进行初始化。首次 Publish 需要在 AgentCore 新建团队空间租户。

首期不提供用户可感知的 Harness 版本管理，但每次 Agent Run 启动时必须固化一份运行配置快照，包括 Agent 角色、模型、Skill、Tool、知识库范围、环境变量引用和 AgentFlow 配置快照。Publish 只是将当前配置同步至 AgentCore，不提供历史版本对比和回滚。

![Harness配置总栏目设置低保真图](./pic/harness.png)

#### 能力中心

能力中心用于配置团队空间内 Agent 可使用的能力集合。

能力中心包括：

1. Skill：Skill 来源于公司 AgentCenter 市场。团队管理员可以从 AgentCenter 中选择已发布版本的 Skill，并绑定到当前团队空间。
2. Tool：Tool 来源于公司 AgentCenter 市场。团队管理员可以从 AgentCenter 中选择已发布版本的 Tool，并绑定到当前团队空间。

未来可能扩展更多的能力中心配置。

---

#### 知识库

知识库用于为团队空间内的 Agent 提供规范、文档、代码等业务知识。

知识库配置页面低保真图：

![知识库配置页面低保真图](./pic/knowledge%20base.png)

知识库由以下部分组成：

1. 团队空间全局 Agent.md。
2. 团队空间文件目录。当前阶段团队空间某个文件目录仅支持绑定 Codehub 仓库目录，未来可扩展为其他媒介。

##### 团队空间全局 Agent.md

全局 Agent.md 是团队空间的全局上下文说明文件。

它用于描述团队空间内 Agent 执行每次必须加载的上下文，例如：

* 团队空间总体目标。
* 业务背景。
* 工作规范。
* 术语说明。
* Agent 执行约束。
* 文件和代码修改规则。
* 默认沟通风格。
* 任务完成标准。

Agent.md 支持在线生成和编辑。系统对 Agent.md 提供长度检查、结构检查和健康度提示，引导其仅包含团队空间最关键的全局上下文；细节内容应在知识库其他文件中承载。

长度检查：强提示阈值 < 3k tokens 或 > 15k tokens；硬上限 30k tokens。

---

### 环境变量

环境变量用于为 Agent 执行过程提供必要的运行参数和配置。

环境变量可用于：

* 配置外部系统地址。
* 配置业务参数。
* 配置工具调用参数。
* 配置不同环境下的执行差异。
* 存储非敏感运行配置。

敏感信息应通过公司统一密钥管理或凭证系统管理，不应直接明文存储在普通环境变量中。

环境变量可以被 AgentRuntime、Skill、Tool 或 AgentFlow Step 调用。

---

### Agent 角色

Agent 角色用于定义某类 Agent 在团队空间中的执行身份和能力范围。

一个 Agent 角色包括：

1. 角色名称。
2. 角色说明。
3. AgentRuntime 类型。
4. 主 Agent 使用模型。
5. 执行方式 低信任权限 / 中信任权限 / 高信任权限。
6. 可使用的 Skill。只能从团队空间能力中心已绑定的 Skill 中选择可用子集。
7. 可使用的 Tool。只能从团队空间能力中心已绑定的 Tool 中选择可用子集。
8. 可见知识库范围。
9. 选择该 Agent 后默认的 prompt 模板。

示例 Agent 角色：

* 全栈工程师 Agent：负责端到端完成产品开发任务。
* 市场分析师 Agent：负责定期分析竞品，转化需求。

团队空间创建后系统自动生成默认 Agent 角色，采用默认 AgentCore、默认模型，默认执行方式，以及全局能力中心中授权的 Skill、Tool 和知识库范围，默认 Agent 角色在 Agent 角色清单中展示，不允许用户编辑。用户只能再新增配置 Agent 角色，也可以不新增自定义角色。

执行方式说明：

* 低信任权限：所有工具执行均需人工确认。
* 中信任权限：Agent 可自动执行低风险工具，高风险工具执行需人工确认。高风险工具包括文件写入、脚本执行、创建 MR。
* 高信任权限：Agent 可自动执行所有工具，无需人工确认。

首版 AgentRuntime 类型、主 Agent 使用模型和执行方式均为系统内置，用户不可在前端配置，也不作展示。

---

### AgentFlow

AgentFlow 是由多个 Step 组成的 Agent 协作流程。

它用于描述单个或多个 Agent 角色如何按步骤完成一个复杂任务。

AgentFlow清单和配置页面低保真图：

![AgentFlow清单页面低保真图](./pic/agentflow%20list.png)

![AgentFlow配置页面低保真图](./pic/agentflow.png)

#### AgentFlow Step

每个 AgentFlow Step 包含：

1. Step 名称。
2. 执行 Agent 角色。
3. 启动 Prompt 模板。
4. Agent 任务完成准出 checklist 和规则（包含是否需要人工确认）。
5. 成功后的下一步流转规则。
6. 失败后的下一步流转规则。

#### AgentFlow 流转规则

AgentFlow 支持基础顺序流转：

```text
Step 1 -> Step 2 -> Step 3 -> Succeeded
```

可扩展支持：

* 条件分支。
* 失败重试。
* 并行步骤。

未来可扩展更多流转规则。

#### AgentFlow Run

AgentFlow Run 是 AgentFlow 的一次实际运行实例。

每次 AgentFlow Run 需要记录：

* 触发来源。
* 使用的 AgentFlow 版本。
* 每个 Step 的执行状态。
* 每个 Step 启动的 Agent Run。
* 每个 Step 的输入和输出。
* 每个 Step 的准出 checklist。
* 人工审批记录。
* 文件变更记录。

### 任务配置

AgentSpace 支持三类 Agent 执行入口：
1. 事件任务：由外部或内部事件触发。
2. 定时任务：由调度计划触发。
3. 临时任务：由团队成员手工即时发起。

事件任务和定时任务属于长期配置,用于定义长期存在的事件触发规则和定时调度规则。事件任务和定时任务不仅用于触发 Agent 执行，也用于将不同业务角色的长期工作诉求配置化。例如需求开发、架构看护、环境巡检、Bug 修复、交付统计等，都可以抽象为团队空间中的任务配置。

临时任务是一种区别于预先配置自动触发执行形式的任务，由团队成员即时发起。

#### 事件任务配置

事件任务配置用于描述当某类事件发生时，系统应该如何触发 Agent 执行。

事件任务配置页面低保真图：

![事件任务配置页面低保真图](./pic/event%20task-setup.png)

事件任务的事件源可以包括两种类型：

1. 外部系统事件：来自需求管理系统、DTS 系统、Codehub 等外部系统的事件。
2. 团队空间内部资源事件：来自团队空间 harness 配置、任务状态或 Agent 执行状态变化的事件。

事件任务配置包含：

1. 任务名称。
2. 任务说明。
3. 事件源类型 外部事件 / 内部事件（首期系统内置，用户无需在前端配置）。
4. 事件对象，例如某个外部系统对象、某个知识库目录、某个文件路径。
5. 事件对象动作类型，例如创建、修改、删除、状态变更。
6. 事件对象过滤条件，例如 FE 标签、FE 所属迭代、FE 优先级、FE 责任人、状态值等。
7. 触发后的执行方式（使用 Agent / AgentFlow）。
8. 绑定的单 Agent 或 AgentFlow。
9. 输入参数映射规则（首期系统内置，用户无需在前端配置）。
10. 事件同时启动的条数限制（系统内置，用户无需在前端配置）。
11. 任务完成准出 checklist 和规则。
12. 任务参与人。
13. 事件任务执行实例自动命名规则（首期系统内置，用户无需在前端配置）。
14. 是否启用。
15. 任务触发是否自动执行，不自动执行则将新进任务置为TODO状态，用户手动从看板点击启动该任务开始执行。需要注意同时启动条数达到上限，新加入的自动执行的事件也会进入 TODO 状态，需要区别这些任务，仅支持配置了人工触发的任务可以用户手动启动，如果任务正在等待系统调度，在前端直观显示告知用户。

事件任务需要支持防循环策略。由某个事件任务执行过程产生的资源变更，默认不再次触发同一个事件任务。例如某事件触发条件为需求设计修改，事件任务执行过程中又修改了需求设计，不再次触发事件。

事件任务可以触发：

* 单 Agent 执行。
* AgentFlow 执行。

示例：

```text
当需求系统中新增高优先级需求时，
自动触发「需求开发 AgentFlow」，
事件完成准出 checklist 为 MR清单和流水线执行通过，经过人工确认需求开发已完成。
```

事件任务配置本身是长期存在的规则。

每次事件命中规则后，系统会生成一条事件任务执行记录。

注意首期仅支持 edevops FE 外部事件源。团队空间内部资源事件仅作为未来扩展，不进入首期配置范围。

#### 定时任务配置

定时任务配置用于描述系统按照固定时间或周期自动触发 Agent 执行。

定时任务配置页面低保真图：

![定时任务配置页面低保真图](./pic/schedule%20task-setup.png)

定时任务配置包含：

1. 任务名称。
2. 任务说明。
3. 调度时间 / 调度周期。
4. 触发后的执行方式（使用 Agent / AgentFlow）。
5. 绑定的单 Agent 或 AgentFlow。
6. 输入的预置prompt。
7. 同时启用的定时任务条数限制（系统内置，用户无需在前端配置）。
8. 任务完成准出 checklist 和规则。
9. 任务参与人。
10. 定时任务执行实例自动命名规则（系统内置，用户无需在前端配置，默认为 任务名称- Agent 执行时间）。
11. 是否启用。

定时任务可以触发：

* 单 Agent 执行。
* AgentFlow 执行。

示例：

```text
每天上午 9 点，
自动触发「运营日报 Agent」，
拉取昨日业务数据并生成分析报告。
```

定时任务配置本身长期存在。

每次到达调度时间后，系统会生成一条定时任务执行记录。

单次执行完成后任务配置继续等待下一次调度。

### 临时任务

临时任务是团队成员手工发起的一次即时 Agent 执行。

临时任务可选发起一次单 Agent 执行或一次 AgentFlow 执行。发起单 Agent 执行时不指定 Agent 角色时使用团队空间默认 Agent 角色。

根据选择 Agent 角色 或 Agentflow，可以为用户在输入框提示 Agent 角色 或 Agentflow 首节点配置的预置 prompt 模版。

新建临时任务前端页面低保真图：

![新建临时任务前端页面低保真图](./pic/new%20temporal%20task.png)

临时任务适用于：

* 临时分析问题。
* 生成一次性文档。
* 查询团队知识库。
* 请求 Agent 修改文件。
* 通过自然语言操作团队空间功能。

临时任务需要记录发起人、输入内容、使用的 Agent / AgentFlow、执行过程、输出结果包括文件变更。

临时任务名称根据用户初始输入内容自动生成。

临时任务不属于事件任务或定时任务，但会出现在任务记录看板中，并进行区分标识。

首期任务参与人用于控制运行态交互权限，在事件任务和定时任务中单独设置；任务参与人可以查看并处理被授权任务实例；普通团队成员默认只能查看公开看板摘要，不能处理 Waiting Human；访客无法设置为任务参与人。

### Agent 执行

Agent 执行是 单 Agent 执行 和 AgentFlow 执行的统称，是 AgentSpace 的核心运行能力。AgentFlow 由多个单 Agent 执行组成。

AgentSpace 本身不自研底层 AgentRuntime，对接公司 AgentCore 云服务实现 Agent Run。

Agent 执行记录清单、详情查看、交互页面低保真图：

![Agent 执行记录清单、详情查看、交互页面低保真图](./pic/agent%20run.png)

#### 单 Agent 执行

单 Agent 执行流程如下：

1. 事件任务、定时任务或临时任务触发执行，分类型按规范自动命名执行的Agent Run名称。
2. 系统读取团队空间 Harness 配置。
3. 系统确定使用的 Agent 角色。
4. 系统组装 Agent 配置、Prompt、知识库、Skill、Tool 和环境变量。
5. 系统将执行请求发送给 AgentCore。
6. 系统通过 SSE 流或轮询方式接收 Agent 执行事件。
7. 系统实时展示 Agent 消息、状态变更、工具调用、文件变更和输出结果。
8. 如命中人工介入节点，则进入 Waiting Human 状态。
9. 人工审批通过后继续执行或完成任务。
10. 执行完成后生成结果、文件变更和审计记录。

#### Agent 执行记录与详情查看

每次 Agent 执行需要记录：

#### 输入信息

* 触发来源：某个事件任务、某个定时任务、某个团队成员发起的临时任务。
* 初始 Prompt。
* 使用的 Agent 角色。
* 使用的模型。
* 可用 Skill。
* 可用 Tool。
* 可见知识库范围。
* 环境变量引用。

#### 过程信息

* 思考过程。
* Plan。
* Skill、Tool、SubAgent 调用。
* 状态事件。
* 错误信息。
* 重试记录。
* 文件读取记录。
* 文件变更记录。

#### 输出信息

* 文本结果。
* 文件变更。

#### 交互信息

* 交互人和行为。

### Human in the loop

Human in the loop 是 AgentSpace 的核心协作机制。

人在 Agent 执行过程中可以承担以下职责：

1. 查看 Agent 执行过程。
2. 查看 Agent 输出。
3. 输入 Agent 指令。
4. 补充 Agent 所需信息。
5. 审查 Agent 文件变更。
6. 确认事件任务或定时任务进入完成态。
7. 确认 AgentFlow Step进入完成态。
8. 在异常情况下终止 Agent 执行。

---

### 执行状态机

AgentSpace 的运行态对象统一使用一套基础状态机。

#### 基础状态

| 状态            | 含义                    |
| ------------- | --------------------- |
| Pending       | 已创建，等待执行              |
| Ready         | 已满足执行条件，等待调度          |
| Running       | Agent 或 AgentFlow 执行中 |
| Waiting Human | 等待人工审批、补充信息或审查        |
| Blocked       | 因依赖、权限、输入或外部系统问题被阻塞   |
| Succeeded     | 执行成功                  |
| Failed        | 执行失败                  |
| Cancelled     | 已取消                   |

#### 看板显示状态

基于底层状态看板使用更业务化的展示名称。

| 看板显示             | 底层状态            |
| ---------------- | --------------- |
| TODO             | Pending 或 Ready |
| Running          | Running         |
| Waiting Approval | Waiting Human   |
| Blocked          | Blocked         |
| Succeeded        | Succeeded       |
| Failed           | Failed          |
| Cancelled        | Cancelled       |
| Done （仅 任务记录看板使用） | Succeeded 或 Cancelled（用户点击停止生成，此类取消标记为会话完成） |

### 看板

AgentSpace 提供三类核心看板：

1. 事件任务执行看板。
2. 定时任务执行看板。
3. 任务记录看板。

#### 事件任务执行看板

事件任务执行看板用于查看事件任务触发后的执行记录。

事件任务执行看板页面低保真图：

![事件任务执行看板页面低保真图](./pic/event%20task-kanban.png)

默认状态包括：

1. TODO。
2. Running。
3. Waiting Approval。
4. Blocked
5. Succeeded。
6. Failed。
7. Cancelled。

每条记录展示：

* 事件任务实例名称。
* 事件任务类型。
* 触发时间。
* 当前状态。
* 任务参与人。
* 关联 Agent Run 或 AgentFlow Run。
* 最后更新时间。

事件任务的执行状态可以支持手工修改：

* TODO -> 启动 -> Running
* Running / Waiting Approval / Blocked -> 关闭 -> Cancelled
* Failed、Cancelled -> 重试 -> Running

#### 定时任务执行看板

定时任务执行看板用于查看定时任务每次调度后的执行记录。

定时任务执行看板页面与事件任务执行看板页面保持一致风格，少一个 TODO 类型状态列。

默认状态包括：

1. Running。
2. Waiting Approval。
3. Blocked
4. Succeeded。
5. Failed。
6. Cancelled。

每条记录展示：

* 定时任务名称。
* 调度周期。
* 本次执行时间。
* 执行次数。
* 当前状态。
* 任务参与人。
* 关联 Agent Run 或 AgentFlow Run。
* 最后更新时间。

定时任务的执行状态可以支持手工修改：

* Running / Waiting Approval / Blocked -> 关闭 -> Cancelled
* Failed、Cancelled -> 重试 -> Running

说明：

定时任务配置本身不进入 Succeeded / Failed / Cancelled。

#### 任务记录看板

任务记录看板以 Agent Run / AgentFlow 粒度查看所有任务执行记录。

支持查看：

* 来自事件任务的 Agent Run。
* 来自定时任务的 Agent Run。
* 来自临时任务的 Agent Run。

AgentFlow 在任务记录看板显示当前运行 Step 启动的 Agent Run 的名称和状态，同时标记所属 AgentFlow。点击进入会话详情后，在详情页展示 AgentFlow Step 进度，可切换查看同一 AgentFlow 下其他 Step 产生的 Agent 会话。

默认状态包括：

1. Running。
2. Waiting Approval。
3. Done。
4. Failed。

每条 Agent Run 展示：

* Agent 角色。
* 来源类型。
* 触发人或触发规则。
* 当前状态。
* 所属 AgentFlow。
* 所属任务执行记录。
* 创建时间。
* 最后更新时间。

## 9. 典型场景

### 产品需求自动开发流

当需求系统中新增一个当前月迭代的FE后，事件任务自动触发 AgentFlow：

1. 产品经理指挥 Product Agent 分析 FE 并生成 FE 设计文档。
2. 架构师指挥 SE Agent 校验基于 FE 变更的 SF 设计文档。
3. 开发指挥 Code Agent 基于 SF 设计文档变更修改代码。
4. Review Agent 自动检查文档和代码变更，检查通过后自动将变更提交到 Codehub。

### 架构文档变更审查

当团队空间知识库中的 docs/sf_registry.md 被修改时，事件任务自动触发 架构文档审查 Agent。

1. 架构文档审查 Agent 分析本次特性变更是否影响系统架构。
2. 架构文档审查 Agent 分析当前系统级 Spec 是否需要调整。
3. 架构文档审查 Agent 分析输出调整计划。
4. 架构师人工确认调整计划。
5. 架构文档审查 Agent 变更系统架构文档和系统级 Spec，提交到 Codehub。

### 定时运营分析

每天上午 9 点，定时任务自动触发 Data Agent：

1. Data Agent 拉取昨日业务数据。
2. Data Agent 生成运营分析报告。
3. Data Agent 自动提交报告并推送给各成员。

### 临时团队协作

团队成员在团队空间中新建临时任务，输入：

```text
帮我分析最近三个需求的共性问题，并给出下一版产品优化建议。
```

系统基于团队空间知识库和授权工具启动 Agent 执行。

Agent 输出分析结果后，团队成员可以继续追问、要求修改，或将结果输出为文件保存至知识库。

## 10. 产品边界与非目标

当前阶段 AgentSpace 的非目标包括：

1. 不做通用个人聊天助手，重点服务团队空间内的任务执行和团队协作。
2. 不替代完整项目管理系统，任务看板主要用于 Agent 执行过程追踪。
3. 不替代 Codehub，Codehub 仍是代码和文件的源系统。
4. 不自研底层 AgentRuntime，对接公司 AgentCore。
5. 不做 AgentCenter 市场能力管理，Skill 和 Tool 的发布、审核、版本管理由 AgentCenter 负责。
6. 不做完整数据权限治理平台，只消费租户和团队空间授权结果。
7. 不直接面向外部客户开放，仅面向公司内部员工。

## 11. 首期开发特性集

为了控制产品复杂度，首期建议聚焦以下能力：

1. 创建和管理团队空间。首版系统初始化两个产品线租户，校验用户归属该两个产品线即可在对应产品线租户下创建团队空间。
2. 配置团队成员和权限。
3. 配置团队空间全局能力中心。
4. 配置知识库和知识库全局 Agent.md。
5. 首期支持配置 Agent 角色的名称、说明、可用 Skill、可用 Tool、可见知识库范围和默认 Prompt 模板。AgentRuntime、模型和执行权限等级由系统内置，不在前端展示。
6. 配置简单顺序型 AgentFlow。
7. 引入 edevops FE 外部系统事件源，支持用户配置 FE 关联事件任务。
8. 配置定时任务。
9. 发起临时任务。
10. 查看 Agent 执行记录。
11. 查看 AgentFlow 执行记录。
12. 支持 Waiting Human 审批节点。
13. 支持文件变更展示和人工审查。
14. 提供事件任务、定时任务和 Agent 执行三类看板。

首期可以暂缓：

1. 多 Harness。
2. 复杂 AgentFlow 并行和循环。
3. AgentFlow 支持子流程。
4. 高级权限策略。
5. 自定义看板状态。
6. 团队空间配置模板市场。
7. 团队空间新建时选择模板。
8. 跨团队空间协作。
9. Harness 版本化管理。
10. 支持执行回放。
11. 支持回退当前轮次 Agent 文件修改。
12. 事件任务和定时任务制定规则自动匹配参与人。
13. 人工介入Welink提醒。
14. 人工介入超时机制。
15. 引入更多的 edevops 事件，覆盖架构师、开发、测试、产品经理日常通用事件。
16. 开发运营看板能力。
17. 支持自动生成项目统计看板。
18. 团队成员角色定义，按角色指定任务参与者和渲染看板。
19. 支持配置团队空间内部资源类型事件。

## 12. 全局一致性规则

1. 所有运行态对象必须关联团队空间。
2. 所有 Agent Run 必须记录触发来源。
3. 所有 Agent Run 必须固化启动时使用的 Agent 角色、模型、Skill、Tool、知识库范围。
4. 所有文件变更必须关联到具体 Agent Run。
5. 所有 Waiting Human 状态必须有明确处理人或处理人范围。
6. 所有人工确认、驳回、终止行为必须记录操作者和时间。
7. 所有任务执行记录必须能够跳转到关联 Agent Run 或 AgentFlow Run。
8. 所有 Agent Run 必须能够查看 Agent 运行记录详情。
9. 所有 AgentFlow Step Run 必须能够查看输入、输出和准出 checklist。
10. 首期 AgentSpace 不自建独立文件源。团队空间知识库中的文件目录来自 Codehub 绑定目录。Agent 产生的文件变更本质上是对 Codehub 绑定目录的变更草稿 / diff，任务完成提交到 Codehub。AgentSpace 只展示文件树、diff、审查状态和关联执行记录，不替代 Codehub 的仓库管理能力。
